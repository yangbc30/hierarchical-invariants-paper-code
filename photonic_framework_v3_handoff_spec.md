# Photonic Jordan Framework v3 ‚Äî Design & Codex Handoff Spec

## 1. Purpose

This document fixes the architecture for the next refactor of the photonic Jordan framework. It is written for two audiences:

1. **The project owner** ‚Äî to review the mathematical/software boundary.
2. **Codex or any coding agent** ‚Äî to implement the next version without changing the public mental model.

The goal is to preserve the successful user-facing API from the v2 demo while rebuilding the internal representation layer so the framework supports:

- global Jordan spaces,
- sector-local Jordan spaces,
- multiplicity/copy-resolved projectors,
- Schur‚ÄìWeyl-aware state representations,
- passive linear-optical evolution,
- invariant computation on global / sector / copy-local hierarchies.

---

## 2. Non-negotiable design decisions

### 2.1 Public API stays simple

The user should mainly interact with:

- `PhotonicSystem`
- `PhotonicState`
- `PhotonicUnitary` (or equivalent unitary wrapper)
- `InvariantReport`

The user should **not** need to manually construct internal engines like filtration builders, projector families, or decomposition objects.

### 2.2 Internal code must be representation-aware

The framework must distinguish between the following representations:

1. **Labeled tensor representation**
   - Hilbert space: `(C^m)^{‚äó n}`
   - Used for explicit basis states, permutation action, raw density matrices.

2. **Schur representation**
   - Hilbert space decomposition:
     `H ‚âÖ ‚äï_Œª (U_Œª ‚äó V_Œª)`
   - `U_Œª`: irrep space for `U(m)` / linear optics
   - `V_Œª`: multiplicity space for `S_n`

3. **Sector-block representation**
   - Block decomposition by partition `Œª`
   - Uses projectors `Q_Œª`

4. **Copy-block representation**
   - Block decomposition inside a fixed `Œª` by multiplicity basis index `a`
   - Uses projectors `Q_{Œª,a}`

5. **Bosonic Fock representation**
   - Only for the fully symmetric sector `(n)`
   - Useful as a convenience representation, not the master representation.

### 2.3 Global / sector / copy-local Jordan spaces are distinct objects

The framework must support three separate filtrations:

- global Jordan filtration,
- `Œª`-sector-local Jordan filtration,
- `(Œª, a)` copy-local Jordan filtration.

These are **not** to be conflated.

### 2.4 Lazy evaluation and caching are mandatory

Heavy structures must be built on demand and cached:

- Schur decomposition
- sector projectors
- copy projectors
- Jordan bases / projectors
- transformed representations of states

The system object owns long-lived caches. State objects may own local caches for transformed forms and derived analyses.

## 3. Mathematical contracts that the code must respect

## 3.1 External reduced state symmetry

For physical reduced external states originating from a permutation-symmetric total state,

`[P_œÉ, œÅ_ext] = 0` for all `œÉ ‚àà S_n`.

This implies `œÅ_ext` is block-diagonal in Schur basis by partition `Œª`.

## 3.2 Passive linear optics acts identically on each particle

If `S` is the single-particle unitary, then the `n`-particle labeled-tensor unitary is

`U = S ‚äó S ‚äó ... ‚äó S`.

Passive linear optics is generated by

`E_st = Œ£_i |s><t|_i`.

## 3.3 Global Jordan filtration

Define `J_{‚â§j}` as the span of operator monomials of degree at most `j` in the generators `E_st`.

Define the exact layer

`ŒîJ_j := J_{‚â§j} ‚äñ J_{‚â§j-1}`

with `J_{‚â§-1} = {0}`.

Projectors:

- cumulative projector: `ŒÝ_{‚â§j}`
- exact-layer projector: `ŒÝ^Œî_j`

Under passive linear optics,

- `R_U(X) = U X U‚ÄÝ` preserves `J_{‚â§j}`
- therefore `[R_U, ŒÝ_{‚â§j}] = 0`
- and also `[R_U, ŒÝ^Œî_j] = 0`

## 3.4 Sector-local Jordan filtration

For a partition `Œª`, define

`E_st^(Œª) := Q_Œª E_st Q_Œª`

and define the local filtration inside the `Œª`-sector by monomials in `E_st^(Œª)`.

Equivalent absorption form:

`Poly^j(E_st^(Œª)) = Poly^j(E_st) Q_Œª`

for operators supported in the `Œª` block.

Local projector notation:

- cumulative: `ŒÝ_{‚â§j}^(Œª)`
- exact-layer: `ŒÝ_j^{Œî,(Œª)}`

These must satisfy support/absorption:

- `Q_Œª O = O Q_Œª = O` for every basis element `O` in the local Jordan space.

Consequences:

- `[ŒÝ_{‚â§j}^(Œª), Q_Œª(¬∑)Q_Œª] = 0`
- same for exact-layer projector.

## 3.5 Multiplicity/copy-local structure

Inside one Schur block,

`H^(Œª) ‚âÖ U_Œª ‚äó V_Œª`

with multiplicity space `V_Œª`.

The generators satisfy

`E_st |_(Œª) = E_st^(Œª,U) ‚äó I_(V_Œª)`

so passive linear optics acts trivially on multiplicity space.

After a multiplicity basis is fixed, define

`Q_{Œª,a} = I_(U_Œª) ‚äó |a><a|`.

Then

`[Q_{Œª,a}, E_st] = 0`

and copy-local Jordan spaces are well-defined.

### Important convention

Objects defined only with `Q_Œª` are **basis-independent**.
Objects defined with `Q_{Œª,a}` are **basis-dependent**, because they require a chosen multiplicity basis.

The code and documentation must state this explicitly.

## 4. Public API target

## 4.1 Canonical user workflow

The intended workflow is:

```python
sys = PhotonicSystem(m_ext=3, n_particles=3, particle_type="boson")

rho = sys.state.from_modes_and_gram([0, 1, 2], gram="indistinguishable")
S = sys.unitary.haar()
rho_out = rho.evolve(S)

print(rho_out.invariant.I_exact(2))
print(rho_out.invariant.I_exact(2, sector=(2, 1)))
print(rho_out.invariant.I_exact(2, copy=((2, 1), 0)))
print(rho_out.analyze(max_order=3))
```

The user should not need to instantiate decomposition or projector builders directly.

## 4.2 Public class sketch

Responsibilities:

- owns immutable model spec,
- owns long-lived caches,
- exposes factories/services:
  - `state`
  - `unitary`
  - `decomposition`

```python

```

Representative user properties:

- `sys.state`
- `sys.unitary`

### `PhotonicState`

Represents a state tied to one system.

Internal fields should include at least:

- `system`
- `data`
- `rep`
- `kind`
- `_cache`

Recommended representation labels:

- `"tensor"`
- `"schur"`
- `"sector_blocks"`
- `"copy_blocks"`
- `"fock"`

Recommended kind labels:

- `"density"`
- `"ket"`

Essential methods:

- `evolve(S_or_unitary)`
- `evolve_haar()`
- `to(rep)`
- `project_sector(lam)`
- `project_copy(lam, a)`
- `project_jordan(order, kind="exact", sector=None, copy=None)`
- `analyze(max_order, sector=None, copy=None)`
- `is_physical()`
- `trace()`
- `purity()`

### `PhotonicUnitary`

A wrapper for single-particle and lifted many-particle passive unitaries.

User entry points should include:

- `sys.unitary.haar()`
- `sys.unitary.from_matrix(S)`
- future: `sys.unitary.beamsplitter(...)`, `mesh(...)`

### `InvariantReport`

A lightweight result object for analysis.

Suggested fields:

- `cumulative`
- `exact`
- `spectra`
- `metadata`

## 6. Representation conventions

## 6.1 One source of truth

The master computational representation should initially remain the labeled tensor representation. Schur and block forms are derived/cached representations.

This keeps the implementation incremental and avoids requiring the entire codebase to switch to Schur-first immediately.

## 6.2 State conversion API

`PhotonicState.to(rep)` should be lazy.

Recommended behavior:

- if the requested representation is already cached, return cached data,
- otherwise compute it once and cache it.

## 6.3 Sector and copy views

The state object should support:

- `rho.blocks()` ‚Üí sector blocks
- `rho.block(lam)` ‚Üí one sector block
- `rho.copy_block(lam, a)` ‚Üí one copy block

These should return either new `PhotonicState` views or lightweight wrappers, but the semantics must be consistent across the package.

## 7. State construction requirements

## 7.1 Required constructors

The state factory must support at least:

### A. External modes + internal Gram matrix

```python
sys.state.from_modes_and_gram(ext_modes, gram)
```

Accepted `gram` inputs:

- `"indistinguishable"` ‚Üí all-ones Gram matrix
- `"distinguishable"` ‚Üí identity Gram matrix
- scalar overlap `x` ‚Üí diagonal 1, off-diagonal `x`
- explicit matrix

### B. Random commutant state

```python
sys.state.random_commutant_state()
```

This should directly sample physical external reduced states that commute with permutations.

### C. Random sector state

```python
sys.state.random_sector_state(lam)
```

### D. Manual density matrix

```python
sys.state.from_density_matrix(mat, rep="tensor")
```

## 7.2 Physical validation

Every constructor returning a density matrix should support validation of:

- Hermiticity
- positive semidefiniteness
- trace one

The validation API should be reusable:

- `rho.is_physical()`
- `rho.validate()`

## 8. Jordan hierarchy API contract

## 8.1 Scope selection

Every Jordan projection/invariant call must support exactly one of the following scopes:

- global scope: default
- sector-local scope: `sector=Œª`
- copy-local scope: `copy=(Œª, a)`

Supplying both `sector` and `copy` at once should raise a clear error.

## 8.2 Projector semantics

Every hierarchy must expose both:

- cumulative projector
- exact-layer projector

No ambiguous naming such as `layer_weight` for cumulative norms.

Recommended invariant names:

- `I_cumulative(j, ...)`
- `I_exact(j, ...)`
- `layer_spectrum(j, ...)`

## 8.3 Basis construction policy

Initial implementation may remain brute-force / prototype-grade, but the API should not assume the construction method.

This allows a later swap to a better representation-theoretic implementation without breaking user code.

## 9. Copy-local/multiplicity conventions

This part must be explicit in code comments and docs.

### 9.1 What is canonical

Canonical / basis-independent:

- partition labels `Œª`
- sector projectors `Q_Œª`
- sector-local Jordan spaces

### 9.2 What depends on a choice

Choice-dependent:

- multiplicity basis labels `a`
- projectors `Q_{Œª,a}`
- copy-local Jordan spaces / invariants

### 9.3 Documentation rule

Any public method involving `copy=(Œª, a)` must mention that the result depends on the currently fixed multiplicity basis convention.

## 10. Testing requirements

## 10.1 Representation tests

Must include tests for:

- permutation matrices form a representation of `S_n`
- `Q_Œª^2 = Q_Œª`
- `Œ£_Œª Q_Œª = I`
- Schur transform `W` is unitary
- `Q_{Œª,a}` are orthogonal projectors resolving `Q_Œª`

## 10.2 Generator and dynamics tests

Must include tests for:

- `[E_st, P_œÉ] = 0`
- generator commutator algebra
- passive evolution preserves trace and positivity
- blockwise passive evolution matches full-tensor evolution

## 10.3 Global Jordan tests

Must include tests for:

- nesting `J_{‚â§j-1} ‚äÇ J_{‚â§j}`
- orthogonality of exact layers
- `[R_U, ŒÝ_{‚â§j}] = 0`
- `[R_U, ŒÝ^Œî_j] = 0`

## 10.4 Local Jordan tests

Must include tests for:

- support absorption in sector-local basis
- `[ŒÝ_{‚â§j}^{(Œª)}, Q_Œª(¬∑)Q_Œª] = 0`
- same for exact layers
- local invariants remain unchanged under passive evolution

## 10.5 Copy-local tests

Must include tests for:

- `Q_{Œª,a}` commute with `E_st`
- copy-local support absorption
- copy-local projector commutation
- blockwise/copywise invariants remain unchanged under passive evolution

## 10.6 API smoke tests

Notebook-style tests should verify the intended high-level usage works with minimal setup.

## 11. Implementation phases

## Phase 1 ‚Äî internal representation refactor

Goal:

- add `SchurWeylDecomposition`
- add sector and copy projectors
- keep v2 public API intact

Deliverables:

- tensor ‚Üî Schur conversion
- `Q_Œª`, `Q_{Œª,a}`
- sector/copy block extraction from states

## Phase 2 ‚Äî local Jordan hierarchies

Goal:

- add sector-local hierarchy
- add copy-local hierarchy
- extend invariant engine to scope-aware calls

Deliverables:

- `project_jordan(..., sector=...)`
- `project_jordan(..., copy=...)`
- `I_exact / I_cumulative / spectra` in all scopes

## Phase 3 ‚Äî cleanup and ergonomics

Goal:

- polish state views
- improve docs
- stabilize naming
- add richer reports

Deliverables:

- report objects
- better repr/printing
- convenience constructors

## Phase 4 ‚Äî future device layer

Deferred from this refactor, but must remain compatible.

Future additions may include:

- beamsplitters
- phase shifters
- meshes
- active Gaussian elements
- nonlinear elements/channels

The public API designed here must not block these.

## 12. Work rules for Codex implementation

This section is written as direct implementation guidance.

1. **Do not rewrite the public API unless explicitly requested.**
2. **Do not collapse global / sector / copy-local Jordan spaces into one ambiguous object.**
3. **Do not treat copy-local objects as basis-independent.**
4. **Keep cumulative and exact-layer semantics separate in both code and naming.**
5. **Preserve notebook ergonomics.** The user-facing workflow must stay short.
6. **Prefer small, reviewable commits.**
7. **Add tests together with each mathematical feature.**
8. **Document every new public method with scope semantics.**

## 15. Final summary

Keep the v2 user experience.
Refactor the internal representation layer.
Introduce explicit Schur/multiplicity structure.
Implement global, sector-local, and copy-local Jordan hierarchies as separate scoped objects.
Treat sector-local quantities as canonical and copy-local quantities as basis-dependent.
Make all heavy structures lazy and cached.
Tie every mathematical feature to tests.

That is the v3 target.
